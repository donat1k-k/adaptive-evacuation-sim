# AGENTS.md

Постоянные правила для AI-агентов в этом проекте. Короткий чек-лист, который агент держит в голове всегда. Подробности — в `docs/PROJECT_SPEC.md` (источник правды) и `docs/PROJECT_PLAN.md` (план). Этот файл их не дублирует.

---

## 1. Project identity

- Проект — **исследовательский стенд адаптивной эвакуации**: моделирование эвакуации и сравнение алгоритмов маршрутизации по метрикам.
- Это **не игра, не ОБЖ, не обычный сайт**.
- Фокус — **информатика**: графы, алгоритмы поиска пути, агентное моделирование, симуляция, метрики, воспроизводимые эксперименты, выводы.
- Ценность проекта — **данные** (метрики, серии, кривая эвакуации), а не картинка. При спорах: корректность и воспроизводимость важнее красоты UI.

## 2. Required reading before major work

Перед любой крупной задачей агент **обязан** прочитать:
- `docs/PROJECT_SPEC.md` — что и почему делаем (модель, алгоритмы, метрики).
- `docs/PROJECT_PLAN.md` — этапы E0–E15, MVP, Strong, эксперименты.
- `docs/HANDOFF.md` — где проект сейчас, что дальше.

## 3. Workflow rules

- Работать **только внутри текущего репозитория**.
- Не переписывать unrelated-файлы.
- Не менять архитектуру без объяснения (структура — `PROJECT_PLAN.md §10`, инварианты — `SPEC §24`).
- Не добавлять зависимости без причины; причину писать в `docs/DECISIONS.md`.
- Сложные изменения **сначала планировать**, потом выполнять.
- После крупного этапа — обновлять `docs/HANDOFF.md`.
- После стабильного этапа (зелёные проверки) — предлагать git commit.
- **Не начинать следующий этап без подтверждения пользователя.**
- Один агент на папку — параллельные автоправки запрещены.

## 4. Scope control (без явного разрешения — запрещено)

backend · auth · база данных · 3D · VR · нейросети · компьютерное зрение · импорт планов из изображений · физическая симуляция огня · сложная психология толпы (паника/стадность/типы агентов) · многоэтажность · мобильное приложение · редактор карт в MVP.

Всё это — Future Work (`SPEC §22`). При соблазне — сверка с §22 и отказ.

## 5. Architecture rules

- Не складывать всю логику в один React-компонент.
- **Симуляционное ядро отдельно от UI** (ядро не импортирует React; прогон headless).
- Алгоритмы отдельно от визуализации, за единым интерфейсом (заменяемы).
- Модели данных типизированы (TypeScript), пассивны (данные, не поведение).
- Сценарии — отдельные данные (JSON/код), не зашиты в логику.
- Метрики считаются автоматически, корректно (время — только по эвакуированным; % эвакуации всегда рядом).
- **Seed и воспроизводимость не ломать**: вся случайность — от seed; конфиг+seed в каждой записи результата.

## 6. MVP rules (держать узким)

- Карты — готовые сценарии/JSON, без редактора.
- 3 алгоритма: **A1** (Nearest Exit), **A2** (Shortest Path A*), **A4** (Adaptive Weighted A*).
- Дискретные тики; один агент на клетку; конфликты — через seed/random-приоритет.
- Базовая (debug-уровня) визуализация; базовые метрики; кривая эвакуации; экспорт CSV/JSON.
- Всё сверх этого — Strong Version или Future Work, не лезет в MVP.

## 7. Quality rules

- После крупных изменений запускать доступные проверки (`tsc --noEmit`, lint, sanity-тесты `SPEC §18`, тесты метрик).
- TypeScript без очевидных ошибок.
- Не оставлять мёртвый/мусорный код.
- Не вводить магические параметры без описания (коэффициенты модели — в `docs/DECISIONS.md`).
- Сложные решения фиксировать в `docs/DECISIONS.md`.
- Текущий статус фиксировать в `docs/HANDOFF.md`.

## 8. Done means

Задача готова, если:
- реализовано именно то, что просили;
- лишних фич не добавлено;
- проект не сломан;
- проверки запущены — или честно сказано, что не запускались и почему;
- `docs/HANDOFF.md` обновлён после крупного этапа;
- пользователь получил краткий отчёт: **что изменено · какие файлы · что проверено · следующий шаг**.
