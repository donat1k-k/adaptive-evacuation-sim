# PROJECT_PLAN.md

**Проект:** Программное моделирование и сравнительный анализ алгоритмов адаптивной маршрутизации при эвакуации людей в динамически изменяющейся среде.

**Статус документа:** план разработки и исследования. Версия 1.0.

**Связанные документы:** `docs/PROJECT_SPEC.md` (источник правды по модели, алгоритмам, метрикам), `docs/HANDOFF.md` (память между сессиями), `docs/DECISIONS.md` (журнал решений), `AGENTS.md` / `CLAUDE.md` (правила для агентов).

---

## 0. Назначение и правила чтения документа

Это **практический план работы**, а не текст исследовательской работы и не дубль спецификации. `PROJECT_SPEC.md` отвечает на вопрос «что и почему мы делаем». Этот документ отвечает на вопрос «в каком порядке, какими этапами и как проверять».

Правила использования:
- Если план противоречит `PROJECT_SPEC.md` — прав `PROJECT_SPEC.md`. Расхождение фиксируется в `DECISIONS.md`, и план правится.
- План задаёт **крупные этапы**, а не микрозадачи. Внутри этапа агент сам дробит работу.
- Этапы выполняются по порядку. Перепрыгивать этап (например, делать красивый UI до ядра) запрещено без явного решения автора.
- Каждый этап завершается проверкой, commit'ом и (для крупных) обновлением `HANDOFF.md`.

## 1. Общая стратегия разработки

**Сначала модель и MVP, потом красота.** Ценность проекта — не картинка, а данные: метрики, серии, кривая эвакуации, выводы. Поэтому сначала строится то, что производит корректные числа, а оформление подключается, когда числам уже можно доверять.

**Почему не начинаем с красивого UI.** Красивый интерфейс на пустом ядре создаёт иллюзию готовности и провоцирует переделки. UI зависит от модели данных и движка; если они меняются, UI переписывается. Сначала стабилизируем ядро, затем рисуем поверх него.

**Почему программа — исследовательский стенд, а не симулятор-игра.** Главный потребитель программы — эксперимент. Архитектура подчинена тому, чтобы симуляцию можно было прогнать **без отрисовки** (headless), много раз, с разными seed, и собрать метрики. Визуализация — вспомогательный слой для отладки, демонстрации и понимания, а не ядро.

**Почему карты сначала как JSON/данные, а редактор позже.** Эксперименту нужны фиксированные, воспроизводимые, версионируемые сценарии. Сценарий-как-данные легко хранить в git, сравнивать, повторять. Редактор карт — это отдельный UI-проект со своей сложностью; он не нужен для научного результата и переносится в Strong Version (и то опционально).

**Главный архитектурный инвариант (из SPEC §24):** ядро симуляции не зависит от React/визуализации. Все этапы соблюдают это правило.

**Последовательность приоритетов при любом споре:**
1. Корректность модели и метрик.
2. Воспроизводимость.
3. Возможность серийных экспериментов (headless).
4. Понятность кода и архитектуры.
5. Удобство UI.
6. Красота UI.

### E0. Подготовка репозитория и документации

- **Не входит:** любой код приложения, установка npm-зависимостей сверх инициализации.
- **Критерии готовности:** репозиторий инициализирован; `docs/` полон; `AGENTS.md`/`CLAUDE.md` содержат правила; первый commit сделан.
- **Файлы/папки:** `.gitignore`, `AGENTS.md`, `CLAUDE.md`, `README.md`, `docs/*`.
- **Риски:** агент начнёт сразу создавать React-проект. → Явно: E0 не трогает `src/`.
- **Проверка:** `git status` чистый; `git log` показывает первый commit; документы открываются.
- **Commit:** `docs: init repo, fill agent rules and project docs`.
- **HANDOFF:** отметить, что репозиторий инициализирован, правила агентов заполнены, следующий этап — каркас.

### E1. Каркас приложения

- **Цель:** минимальный запускаемый Vite + React + TypeScript проект с целевой структурой папок.
- **Реализовать:** инициализация Vite (react-ts); целевая структура `src/` (см. §10) с пустыми index-файлами и короткими комментариями о назначении папки; настройка TypeScript (strict), ESLint; «Hello, стенд» в `App.tsx`.
- **Не входит:** любая модель, движок, алгоритмы, графики; UI-библиотеки; state-менеджер (пока не нужен).
- **Критерии готовности:** `npm run dev` запускает пустую страницу; `npm run build` проходит; `tsc --noEmit` без ошибок; структура папок создана.
- **Файлы/папки:** `package.json`, `vite.config.ts`, `tsconfig.json`, `src/{models,simulation,algorithms,scenarios,components,metrics,experiment,export,utils}/`, `src/App.tsx`, `src/main.tsx`.
- **Риски:** лишние зависимости «на будущее»; преждевременный выбор UI-kit. → Ставить только Vite/React/TS; остальное по мере необходимости с записью причины в `DECISIONS.md`.
- **Проверка:** dev-сервер поднимается; сборка зелёная; типчек зелёный.
- **Commit:** `chore: scaffold vite+react+ts, create module structure`.
- **HANDOFF:** каркас готов, структура папок зафиксирована, стек подтверждён.

### E2. Модель данных (`src/models`)

- **Цель:** типы и структуры данных для всей программы — единый словарь, на который опираются движок, алгоритмы, метрики.
- **Реализовать (по SPEC §5–9, §16):** типы `Cell`/координаты; `EnvironmentModel` (сетка `W×H`, стены, выходы, стартовые зоны, блокировки); `Agent` (поля и состояния из §6); `Scenario` (карта + агенты + события + параметры); `SimulationConfig` (`{сценарий, алгоритм, параметры, seed}`); `SimulationResult` / структуры метрик; `DynamicEvent` (`{тип, цель, тик}`). Чистые типы и простые фабрики, без логики симуляции.
- **Не входит:** алгоритмы поиска пути; движок тиков; отрисовка; вычисление метрик (только их типы).
- **Критерии готовности:** типы компилируются; можно сконструировать пустую среду, агента, сценарий и конфиг в коде/тесте; поля соответствуют SPEC §6, §15, §16.
- **Файлы/папки:** `src/models/*.ts`, `src/utils/` (геометрия: манхэттен, соседи клетки).
- **Риски:** «утечка» логики в модели (модель должна быть данными, а не поведением); расхождение полей с SPEC. → Держать модели пассивными; сверять поля по SPEC §6.
- **Проверка:** `tsc --noEmit`; мини-тест на создание объектов; ручная сверка полей агента/конфига с SPEC.
- **Commit:** `feat(models): data types for environment, agent, scenario, config, result`.
- **HANDOFF:** модель данных зафиксирована; перечислить ключевые типы и их назначение.

### E3. Симуляционный движок (`src/simulation`)

- **Цель:** ядро — синхронная пошаговая симуляция с разрешением конфликтов за клетку. Самый ответственный этап: без корректного контеншена сравнение алгоритмов бессмысленно.
- **Реализовать (по SPEC §7–8, §15):** `SimulationEngine` с пошаговым `step()`; фаза намерений → фаза разрешения конфликтов через seed-приоритет; правило «одна клетка — один агент»; обработка проигравших (ждут/помечаются); застревание (`N_stuck` → `stuck`/`blocked`); горизонт `T_max`; единый ГПСЧ от `seed` (детерминизм); вычисление плотности в окрестности. Движок работает **headless** — без отрисовки. Маршрутизация подаётся как подключаемая стратегия (интерфейс алгоритма), сам алгоритм — в E5.
- **Не входит:** конкретные алгоритмы (заглушка/простейший построитель пути); визуализация; метрики (кроме нужного движку); динамические события (E6).
- **Критерии готовности:** движок прогоняет сценарий от старта до `T_max`/полной эвакуации headless; детерминизм по seed (два прогона идентичны); ни один агент не в стене, не дублируется, не делит клетку; число агентов сохраняется.
- **Файлы/папки:** `src/simulation/SimulationEngine.ts`, `src/simulation/rng.ts` (seedable ГПСЧ), `src/simulation/conflict.ts`, `src/utils/`.
- **Риски:** результат зависит от порядка обхода агентов (нарушение синхронности); скрытая недетерминированность (`Math.random`, время, обход Map/Set). → Только seedable ГПСЧ; двухфазное разрешение; тест детерминизма обязателен.
- **Проверка:** sanity-тесты SPEC §18 п.2,3,4,5 (сохранение агентов, стены, эксклюзивность клетки, детерминизм); один агент на пустой карте доходит за манхэттен-число шагов.
- **Commit:** `feat(simulation): synchronous tick engine with seeded conflict resolution`.
- **HANDOFF:** ядро движка готово и детерминировано; зафиксировать решение по «движению цепочкой» (SPEC §8 п.5) в `DECISIONS.md`.

### E4. Базовая визуализация (`src/components`)

- **Цель:** минимальная отрисовка состояния симуляции для отладки и понимания. Намеренно некрасивая.
- **Реализовать:** Canvas-рендер сетки (стены, выходы, старты, агенты, блокировки); кнопки запуск / пауза / сброс / шаг; ускоренный прогон (скорость тиков); счётчик тика и числа эвакуированных. Визуализация **читает** состояние движка и не содержит логики симуляции.
- **Не входит:** графики метрик (E11); красивый дизайн; темы; анимации; heatmap (Strong); редактор (Strong).
- **Критерии готовности:** видно, как агенты идут к выходам и эвакуируются; контролы работают; смена сценария перерисовывает; отрисовка не влияет на результат headless-прогона.
- **Файлы/папки:** `src/components/SimulationCanvas.tsx`, `src/components/Controls.tsx`, `src/App.tsx`.
- **Риски:** логика симуляции «протекает» в компонент; UI становится источником правды о состоянии. → Компонент только рендерит и шлёт команды движку; состояние живёт в движке/сторе.
- **Проверка:** визуально совпадает с headless-метриками (одинаковый seed → одинаковое поведение); инвариант «ядро не зависит от React» не нарушен (движок собирается без импортов React).
- **Commit:** `feat(ui): debug canvas renderer and run controls`.
- **HANDOFF:** есть видимая симуляция; подчеркнуть, что это debug-слой, не финальный UI.

### E5. Алгоритмы A1 / A2 / A4 (`src/algorithms`)

- **Цель:** три различимых алгоритма маршрутизации за единым интерфейсом, заменяемые без переписывания движка (SPEC §10–11, §24).
- **Реализовать:** общий интерфейс алгоритма (выбор выхода + построение/пересчёт маршрута); A1 Nearest Exit (геометрически ближайший выход, путь BFS/A*, выход не пересматривается); A2 Shortest Path A* (выход по минимальному пути в графе, пересчёт при разрыве); A4 Adaptive Weighted A* (стоимость `base + α·density + δ·exitLoad` для MVP, ∞ для блокировок, пересчёт по триггерам §11, счётчик `reroutes`, гистерезис/период `K`).
- **Не входит:** A3 (Strong); слагаемые `danger`/`smoke` (Strong); подбор финальных коэффициентов (этап экспериментов).
- **Критерии готовности:** каждый алгоритм водит агентов до выходов; A1 не меняет выход при перегрузке; A2 учитывает стены и пересчитывает при разрыве; A4 избегает плотности и считает пересчёты; все три прогоняются на одном сценарии и дают разное поведение.
- **Файлы/папки:** `src/algorithms/{nearestExit,shortestPathAStar,adaptiveWeightedAStar}.ts`, `src/algorithms/pathfinding.ts` (общий A*/BFS), `src/algorithms/index.ts` (реестр).
- **Риски:** A2 и A4 неотличимы (A4 без реального учёта плотности); общий A* зашит под один алгоритм; недопустимая эвристика ломает оптимальность. → Эвристика = манхэттен × base (допустима); тест «на пустой карте все дают длину BFS»; явный тест, что A4 отклоняется от A2 при плотности.
- **Проверка:** sanity SPEC §18 п.1 (оптимальность на пустой карте); поведенческий тест различия A2/A4; реестр позволяет переключать алгоритм в конфиге.
- **Commit:** `feat(algorithms): A1 nearest-exit, A2 shortest-path A*, A4 adaptive weighted A*`.
- **HANDOFF:** три алгоритма работают и различимы; отметить, что коэффициенты α/δ пока черновые (подбор — в экспериментах).

### E6. Динамические события (`src/simulation`)

- **Цель:** динамика среды — без неё проект вырождается в статический поиск пути (SPEC §9).
- **Реализовать:** применение `DynamicEvent` по тику срабатывания; MVP-минимум — блокировка выхода/прохода в заданный тик (клетка → непроходима); реакция движка и алгоритмов (A2/A4 пересчитывают, A1 — только при физическом разрыве пути); пометка застрявших, если выхода больше нет.
- **Не входит:** задымление/зоны риска и веса `danger`/`smoke` (Strong); несколько событий в одном сценарии (Strong); изменение проходимости туда-обратно (Strong).
- **Критерии готовности:** после тика блокировки через выход никто не эвакуируется; пересчитывающие алгоритмы перестраивают маршрут; неперестраивающий A1 показывает деградацию/застревание.
- **Файлы/папки:** `src/simulation/events.ts`, правки `SimulationEngine.ts`, типы в `src/models`.
- **Риски:** событие меняет среду, но кэш пути не инвалидируется (агенты идут в стену); событие недетерминировано по тику. → Инвалидация маршрутов при изменении среды; событие привязано к тику, не к таймеру.
- **Проверка:** sanity SPEC §18 п.6 (реакция на блокировку); тест, что A1 застревает, а A4 выходит через альтернативный выход.
- **Commit:** `feat(simulation): dynamic exit/corridor blocking events`.
- **HANDOFF:** динамика есть; описать, какое событие реализовано и как его задавать в сценарии.

### E7. Сбор метрик (`src/metrics`)

- **Цель:** автоматический, корректный сбор обязательных метрик (SPEC §13–14). Никакой оценки «на глаз».
- **Реализовать:** `MetricsCollector`, считающий по ходу/в конце прогона: makespan, среднее/макс время эвакуации (только по эвакуированным!), процент эвакуированных, число застрявших, нагрузку на каждый выход, число пересчётов (сумма и среднее), среднюю/макс плотность в критических зонах; данные для кривой эвакуации (доля эвакуированных по тикам).
- **Не входит:** желательные метрики Strong (длина маршрута, конфликты, индекс перегрузки, время вычисления, разброс); агрегация по сериям (E9); отрисовка графиков (E11).
- **Критерии готовности:** прогон выдаёт полный набор обязательных метрик; время считается **только по эвакуированным**; процент эвакуации всегда присутствует рядом; кривая эвакуации доступна как данные.
- **Файлы/папки:** `src/metrics/MetricsCollector.ts`, типы результата в `src/models`.
- **Риски (критично):** алгоритмы работают, но метрики неверны — самый опасный тихий баг проекта. Частые ошибки: неэвакуированные попадают в среднее время; makespan считается по `T_max`, а не по последнему вышедшему; double-count выходов. → Юнит-тесты метрик на синтетических прогонах с заранее известным ответом; сверка с правилами SPEC §14.
- **Проверка:** unit-тест метрик на «ручном» сценарии (известные тайминги → известные числа); проверка, что 0% эвакуации не даёт «хорошее» среднее время.
- **Commit:** `feat(metrics): mandatory metrics with correct non-evacuated handling`.
- **HANDOFF:** метрики считаются и протестированы; явно отметить правило «время только по эвакуированным».

### E8. Готовые сценарии S1 / S2 / S3 (`src/scenarios`)

- **Цель:** воспроизводимые сценарии-как-данные (SPEC §17), без редактора.
- **Реализовать:** S1 простой план (комнаты, 2 выхода, без динамики, baseline); S2 узкое место (коридор/дверь → пробка); S3 динамическая блокировка (выход/коридор блокируется во время эвакуации). Каждый — типизированные данные (`Scenario`) с картой, стартовыми зонами, событиями и параметрами; загрузчик/реестр сценариев; выбор сценария в UI.
- **Не входит:** S4/S5 (Strong); редактор карт; генерация случайных карт.
- **Критерии готовности:** три сценария загружаются и прогоняются всеми тремя алгоритмами; S2 действительно создаёт пробку; S3 действительно срабатывает блокировкой; карты валидируются (выход достижим хотя бы для части агентов).
- **Файлы/папки:** `src/scenarios/{s1,s2,s3}.ts`, `src/scenarios/index.ts`, валидатор сценария в `src/utils`.
- **Риски:** «декоративные» карты, где пробка/блокировка не наступает (нет исследовательского контраста); невалидная карта (выход замурован). → Валидатор достижимости; быстрый прогон-проверка, что в S2 растёт плотность, в S3 есть застрявшие у A1.
- **Проверка:** прогон каждого сценария × каждого алгоритма без падений; ручная проверка ожидаемого эффекта (пробка/блокировка).
- **Commit:** `feat(scenarios): S1 baseline, S2 bottleneck, S3 dynamic blocking`.
- **HANDOFF:** три сценария готовы; кратко описать назначение каждого.

### E9. Сравнение алгоритмов и парный прогон по seed

- **Цель:** честное парное сравнение алгоритмов на идентичных входах (SPEC §15).
- **Реализовать:** для фиксированного seed генерируются одни стартовые позиции, затем на них прогоняются все три алгоритма (paired design); сбор метрик по каждому; запуск «сценарий × алгоритм × seed» headless; агрегация по набору seed (mean ± std). Это основа `ExperimentRunner` (полноценно — в E12).
- **Не входит:** большие серии 30–100 (E12); статистические тесты (Strong); UI серий (минимально достаточно).
- **Критерии готовности:** при одном seed все алгоритмы стартуют из идентичных позиций; метрики сравнимы; маленькая серия (≥10 seed) даёт mean ± std на условие.
- **Файлы/папки:** `src/experiment/pairedRun.ts`, `src/experiment/aggregate.ts`, `src/metrics`.
- **Риски:** алгоритмы тайно получают разные старты (нарушен paired design → сравнение нечестное); seed «протекает» между прогонами. → Генерация стартов один раз на seed, затем переиспользование; тест: одинаковый seed → одинаковые стартовые позиции для всех алгоритмов.
- **Проверка:** тест идентичности стартов между алгоритмами при одном seed; ручная сверка, что A1/A2/A4 на одном seed дают разные метрики, но один старт.
- **Commit:** `feat(experiment): paired per-seed algorithm comparison + aggregation`.
- **HANDOFF:** парное сравнение работает; зафиксировать, что сравнение алгоритмов всегда paired.

## 4. План работы с AI-агентами

Основной инструмент сейчас — **Claude Code** (есть лимиты). Структура совместима и с **Codex** — роли описаны так, чтобы подключение Codex не ломало процесс. Базовые правила — в SPEC §27 и `AGENTS.md`/`CLAUDE.md`.

**Claude Code как архитектор** — когда задача требует решений и проектирования: выбор структуры модуля, интерфейс алгоритма, дизайн движка, форма данных метрик, дизайн серии экспериментов, разбор тонких багов (детерминизм, контеншен, корректность метрик), ревью. Здесь — Plan Mode перед изменениями.

**Claude Code как исполнитель** — когда план уже утверждён: написать модуль по согласованному интерфейсу, добавить сценарий, реализовать экспорт, написать тесты. Чёткое ТЗ → прямое выполнение.

**Когда подключать Codex (позже).** После того как архитектура и интерфейсы стабильны (примерно с E5+): Codex берёт рутинную реализацию по утверждённому плану (компоненты, тесты, повторяющийся код), Claude Code остаётся на архитектуре/ревью/сложных решениях. До стабилизации интерфейсов — лучше один инструмент.

**Почему нельзя два агента одновременно на автоправки.** Два агента в одной папке параллельно создают конфликтующие правки, перезатирают файлы и расходятся с `HANDOFF.md` — состояние становится непредсказуемым. Правило: **в один момент времени одну папку правит один агент**. Разделение труда — последовательное (один спланировал → другой выполнил), не параллельное.

**Когда начинать новую сессию.** При смене крупного этапа; когда контекст переполнен и ответы деградируют; после крупного commit'а. Новая сессия начинается с чтения `PROJECT_SPEC.md` + `HANDOFF.md` (+ при необходимости `PROJECT_PLAN.md`, `DECISIONS.md`).

**Как использовать `HANDOFF.md`.** Это память между сессиями. Перед работой — прочитать. После крупного этапа — обновить: где проект, что сделано, что дальше, чего не делать. Коротко и актуально; детали — в SPEC/DECISIONS.

**Как использовать git commit как сейвпоинты.** Commit после каждого стабильного этапа (зелёные проверки) — точка отката. Перед рискованным изменением — commit текущего рабочего состояния. Сообщения по conventional commits. Контрольные точки (MVP) — тегировать.

## 6. План MVP v1 (детально)

База — SPEC §20. MVP — первая **исследовательски полезная** версия, не обязана быть красивой.

**Входит:**
- Vite + React + TypeScript, целевая структура папок.
- Сетка-карта: стены, выходы, стартовые зоны; агенты; синхронная визуальная симуляция.
- Управление: запуск/пауза/сброс/шаг, ускоренный прогон.
- Модель движения с разрешением конфликтов за клетку.
- Одно динамическое событие (блокировка выхода/прохода).
- 3 алгоритма: A1, A2, A4.
- 3 сценария: S1, S2, S3 (как данные).
- Обязательные метрики (§13) с корректной обработкой неэвакуированных (§14).
- Seed и воспроизводимость; серия ≥10 seed на условие; парное сравнение.
- Кривая эвакуации + таблица метрик в UI.
- Экспорт CSV/JSON.
- Sanity-тесты §18.

**Не входит (→ Strong/Future):** A3; S4/S5; задымление/зоны риска; несколько событий; heatmap; редактор карт; желательные метрики §13; серии 30–100; статтесты; 3D/backend/нейросети/психология.

**Минимальные сценарии:** S1 baseline, S2 узкое место, S3 динамическая блокировка.

**Минимальные алгоритмы:** A1, A2, A4 (стоимость A4 = `base + α·density + δ·exitLoad`, ∞ для блокировок).

**Минимальные метрики:** makespan; среднее/макс время (только эвакуированные); % эвакуации; число застрявших; нагрузка по выходам; число пересчётов; средняя/макс плотность.

**Минимальные проверки (sanity §18):** оптимальность на пустой карте; сохранение агентов; непроходимость стен; эксклюзивность клетки; детерминизм по seed; реакция на блокировку. Плюс юнит-тест корректности метрик.

**Критерий «MVP готов»:**
1. Все три сценария прогоняются всеми тремя алгоритмами без падений.
2. Метрики корректны (sanity-тесты §18 зелёные, тест метрик зелёный).
3. Детерминизм: один и тот же `{сценарий, алгоритм, параметры, seed}` → идентичные метрики.
4. Парное сравнение: один seed → идентичные старты для всех алгоритмов.
5. Серия ≥10 seed даёт mean ± std и экспортируется в CSV/JSON с полным конфигом.
6. Кривая эвакуации и таблица метрик отображаются и совпадают с экспортом.
7. Производительность: стабильно 50–200 агентов на сетке до ~40×40.

При выполнении — тег `v1-mvp`, крупное обновление `HANDOFF.md`. Можно начинать E12.

## 8. План экспериментов

Текст работы пока не пишем — фиксируем дизайн (база SPEC §15–17).

**Серии (один фактор на серию):**
- Серия 1 — A1/A2/A4 на простом плане S1 (baseline).
- Серия 2 — влияние числа агентов N (S1/S2).
- Серия 3 — влияние тика блокировки (S3, варьируется тик события).
- Серия 4 — влияние ширины прохода/числа выходов (S2, варианты).
- Серия 5 (Strong) — перегрузка ближайшего выхода / распределение стартов (S5, A3 vs A1).

**Что меняем (независимые):** алгоритм; N; сценарий; тик блокировки; ширина прохода/число выходов.

**Что фиксируем (контролируемые):** карта, набор seed, `T_max`, правила движения, параметры алгоритма кроме исследуемого.

**Seed:** все случайные решения из одного ГПСЧ от целого seed; один seed → детерминированный прогон; парное сравнение — одни старты на seed для всех алгоритмов; серия = набор seed `{s1..sn}`, агрегация mean ± std; ≥10 seed (желательно 30–100).

**Честность сравнения:** только paired design (идентичные старты); сравнивать makespan лишь при равном % эвакуации (иначе сначала %, потом время — SPEC §14); приводить разброс и n.

**Графики потом:** кривая эвакуации (главный); зависимость среднего времени и % эвакуации от N; влияние тика блокировки на число застрявших; (Strong) heatmap плотности, графики с доверительными интервалами.

**Таблицы потом:** сводная по алгоритмам (метрики §13, mean ± std, n); таблицы по сериям 2–4 (метрика от фактора); таблица «эффективность относительно baseline A1».

## 10. Целевая структура репозитория

```
adaptive-evacuation-sim/
├── docs/                  # документация (SPEC, PLAN, HANDOFF, DECISIONS, TODO, RESULTS, PAPER_OUTLINE)
├── results/               # экспортированные результаты экспериментов (CSV/JSON, графики) — вне src
├── src/
│   ├── models/            # типы данных: среда, агент, сценарий, конфиг, результат (пассивные данные)
│   ├── simulation/        # ядро: движок тиков, движение, разрешение конфликтов, события, ГПСЧ. Без React
│   ├── algorithms/        # алгоритмы маршрутизации A1/A2/A4(/A3) за единым интерфейсом + общий A*/BFS
│   ├── scenarios/         # готовые сценарии как данные (S1..S5) + валидатор/реестр
│   ├── metrics/           # сбор и агрегация метрик (корректная обработка неэвакуированных)
│   ├── experiment/        # прогон серий, seed-менеджмент, парное сравнение, агрегация (headless)
│   ├── export/            # экспорт CSV/JSON с полным конфигом
│   ├── components/        # UI: Canvas-рендер, контролы, графики, таблицы (только отрисовка)
│   └── utils/             # геометрия, валидация, вспомогательное
├── AGENTS.md              # общие правила для агентов
├── CLAUDE.md              # правила для Claude Code
├── README.md
└── (vite.config.ts, tsconfig.json, package.json, .gitignore)
```

**Назначение папок:**
- `models` — единый словарь данных; пассивные типы, без поведения. На них опираются все.
- `simulation` — сердце стенда; детерминированный headless-движок. **Не импортирует React.**
- `algorithms` — заменяемые стратегии маршрутизации за общим интерфейсом; сравниваются без переписывания движка.
- `scenarios` — воспроизводимые входные данные эксперимента; версионируются в git.
- `metrics` — превращает прогон в числа; самый критичный по корректности модуль.
- `experiment` — оркестратор серий и seed; основной потребитель ядра при исследовании.
- `export` — мост к внешнему анализу и воспроизводимости.
- `components` — визуальный слой; только читает состояние и шлёт команды, логики не содержит.
- `utils` — общие мелочи (манхэттен, соседи, валидация карты).
- `docs` / `results` — знание о проекте и данные экспериментов отдельно от кода.

## 11. Правила качества

- **Не писать всё в одном файле.** Логика разнесена по модулям §10; вся логика не свалена в один компонент (SPEC §25).
- **Не менять архитектуру без объяснения.** Любое отклонение от структуры §10 / инвариантов SPEC §24 — с обоснованием в `DECISIONS.md` и (для крупного) согласованием с автором.
- **Не добавлять зависимости без причины.** Каждая новая зависимость — с записью причины в `DECISIONS.md`; одна chart-библиотека; стоп-лист SPEC §22 соблюдается.
- **Инвариант headless.** `simulation`/`algorithms`/`metrics`/`experiment` не зависят от React; ядро прогоняется без отрисовки.
- **После этапа — запускать проверки.** `tsc --noEmit`, lint, sanity-тесты §18, тесты метрик. Красный этап не закрывается.
- **После стабильного этапа — commit.** Зелёные проверки → commit-сейвпоинт; контрольные точки тегировать.
- **После крупного этапа — обновлять `HANDOFF.md`.** Где проект, что сделано, что дальше, чего не делать.
- **Сохранять воспроизводимость.** Вся случайность — от seed; каждая запись результата несёт полный конфиг+seed; round-trip воспроизводимость проверяется.
- **Один агент на папку.** Параллельные автоправки запрещены (§4).
