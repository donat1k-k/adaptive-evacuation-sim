# PROJECT_SPEC.md

**Проект:** Программное моделирование и сравнительный анализ алгоритмов адаптивной маршрутизации при эвакуации людей в динамически изменяющейся среде.

**Статус документа:** главный источник правды (single source of truth). Версия 1.0.

---

## 1. Назначение документа

Это главный документ проекта. Он существует не для итоговой бумажной работы, а для разработки и сохранения контекста между сессиями с AI-агентами (Claude Code, Codex, ChatGPT и др.).

Документ фиксирует: что создаётся, зачем, по какой научной и технической логике, что должно и не должно получиться, какие требования обязательны, что переносится на потом, какие риски контролируются.

Любой новый агент или человек должен суметь понять проект целиком, прочитав только этот файл, без истории чата.

Что НЕ является этим документом: готовый текст исследовательской работы, план разработки по задачам (см. `PROJECT_PLAN.md`), журнал решений (см. `DECISIONS.md`), память между сессиями (см. `HANDOFF.md`).

## 2. Тема и общий замысел проекта

**Полная формулировка темы:** Программное моделирование и сравнительный анализ алгоритмов адаптивной маршрутизации при эвакуации людей из здания в условиях динамически изменяющейся среды.

**Короткая формулировка:** Исследование алгоритмов адаптивной эвакуации на основе программного моделирования.

**Ключевая идея.** Программа — не самоцель, а исследовательский стенд. С его помощью моделируется эвакуация, сравниваются алгоритмы маршрутизации, собираются количественные данные, строятся графики и делаются выводы.

Проект должен читаться не как «я сделал симулятор», а как: «я разработал программную модель, реализовал несколько алгоритмов маршрутизации, провёл серию вычислительных экспериментов и сравнил эффективность подходов по количественным метрикам».

## 3. Исследовательская рамка

**Объект исследования:** алгоритмы маршрутизации агентов в динамически изменяющейся среде.

**Предмет исследования:** эффективность этих алгоритмов по количественным метрикам эвакуации при разных условиях среды.

**Главный исследовательский вопрос:** при каких условиях адаптивная маршрутизация оказывается эффективнее простых стратегий эвакуации?

**Подвопросы:**
- Когда выбор геометрически ближайшего выхода становится неэффективным?
- Как перегрузка выхода влияет на общее время эвакуации?
- Насколько важна динамическая перестройка маршрута при блокировках?
- Как плотность агентов влияет на скорость эвакуации?
- При каком числе агентов адаптивный алгоритм начинает давать заметное преимущество?
- В каких условиях сложный алгоритм НЕ даёт существенного выигрыша?

**Рабочая гипотеза (фальсифицируемая).** Адаптивные алгоритмы дают статистически значимое преимущество по времени и проценту эвакуации в условиях высокой плотности, динамических блокировок и перегрузки выходов. В простых условиях (низкая плотность, отсутствие динамики) их преимущество исчезает или становится незначимым, а накладные расходы на пересчёт могут ухудшать результат.

Сильный результат проекта — найти и количественно показать точку, где адаптив перестаёт выигрывать. Вывод «адаптив всегда лучший» считается слабым и нежелательным.

## 4. Позиционирование: чем проект является и чем не является

**Проект является** исследованием по информатике: графы, алгоритмы поиска пути, агентное моделирование, динамические веса, вычислительный эксперимент, метрики, статистика, воспроизводимость.

**Проект НЕ является** и не должен выглядеть как: игра с человечками; обычный сайт без научной части; визуализация «для красоты»; работа по ОБЖ; социальная работа про безопасность; нейросетевой проект «ради ИИ»; 3D-демонстрация без метрик; профессиональный расчёт МЧС.

**В фокусе — информатика.** Правила эвакуации, нормативы, психология паники, физика огня, датчики/камеры/дроны допускаются только как краткий контекст во введении, но не как содержание работы.

**Анти-цель формулируется как принцип:** не делать «красивый симулятор ради симулятора». Делать: исследовательский стенд + динамическая среда + сравнение алгоритмов + метрики + эксперименты + выводы. Этот принцип имеет приоритет при любых спорных решениях по разработке.

## 5. Модель среды: 2D-сетка как граф

Среда — двумерная прямоугольная сетка `W × H` клеток, трактуемая как граф.

| Понятие модели | Граф/сетка |
|---|---|
| Клетка сетки | вершина графа |
| Переход между соседними клетками | ребро графа |
| Стена | непроходимая клетка (вершина исключена) |
| Выход | целевая клетка |
| Стартовая зона | область появления агентов |
| Опасная зона / задымление | клетка с повышенной стоимостью прохода (Strong Version) |
| Блокировка | клетка или выход, ставшие непроходимыми во времени |

**Соседство:** 4-связность (вверх/вниз/влево/вправо) в MVP. 8-связность — Future Work (требует аккуратной обработки диагональных проходов между углами стен).

**Координаты:** целочисленные `(x, y)`, `0 ≤ x < W`, `0 ≤ y < H`.

**Почему такая модель.** Реализуема в рамках школьного проекта; прозрачно объясняется через графы; позволяет напрямую применять алгоритмы поиска пути; не требует физической модели; пригодна для серийных экспериментов.

## 6. Модель агента

Агент — условный человек в модели. Цель модели агента — не психологическая реалистичность, а возможность сравнивать алгоритмы маршрутизации в одинаковых условиях.

Поля агента:
- `id` — уникальный идентификатор;
- `start` — стартовая позиция;
- `pos` — текущая позиция;
- `targetExit` — выбранный выход;
- `route` — текущий маршрут (последовательность клеток);
- `state` — состояние;
- `tStart` — тик начала движения;
- `tEvacuated` — тик эвакуации (если эвакуирован);
- `reroutes` — счётчик пересчётов маршрута;
- `algorithm` — алгоритм, по которому движется агент.

Состояния агента: `waiting` (ожидает), `moving` (движется), `evacuated` (вышел), `blocked` (нет доступного маршрута), `stuck` (застрял — длительно не может продвинуться).

Психология (паника, стадность, индивидуальная информированность, разные типы агентов) в MVP не моделируется. Это Future Work и не входит в ядро.

## 7. Модель времени

Время дискретное, измеряется в **тиках** (шагах симуляции).

- 1 тик = 1 синхронная попытка хода для всех агентов.
- Скорость агента = 1 клетка за тик (агент за тик перемещается максимум на одну соседнюю клетку или остаётся на месте).
- Все временные метрики выражаются в тиках. Перевод в «секунды» не делается: тик — условная единица модельного времени, и для сравнения алгоритмов этого достаточно.
- Симуляция имеет горизонт `T_max` (максимальное число тиков). По достижении `T_max` симуляция останавливается; не вышедшие агенты считаются неэвакуированными (см. раздел 14).

Симуляция **синхронная**: на каждом тике сначала все агенты формируют намерение, затем намерения разрешаются одновременно (см. раздел 8). Это исключает зависимость результата от порядка обхода агентов в коде.

## 8. Модель движения и разрешения конфликтов

Это ядро модели: именно здесь возникают очереди, пробки и плотность. Без корректного контеншена сравнение алгоритмов теряет смысл.

**Правила:**

1. **Одна клетка — один агент.** В любой момент в клетке не более одного агента. Это создаёт физическую конкуренцию за пространство.

2. **Фаза намерения.** На каждом тике каждый движущийся агент по своему маршруту выбирает желаемую следующую клетку `next`. Если `next` занята другим агентом, который сам собирается двигаться, агент всё равно заявляет намерение (продвижение цепочкой допускается, см. п. 5).

3. **Фаза разрешения конфликтов.** Если на одну свободную клетку претендуют несколько агентов, конфликт разрешается через приоритет, определяемый ГПСЧ с фиксированным `seed`. Один агент-победитель занимает клетку, остальные остаются на месте.

4. **Проигравшие агенты** остаются на месте на этом тике. По правилам своего алгоритма они могут пересчитать маршрут (для адаптивного — с учётом возросшей плотности, см. раздел 11) или ждать.

5. **Движение цепочкой (опционально для MVP, обязательно для Strong).** Если агент A хочет в клетку, которую на этом же тике освобождает агент B (B уходит дальше), такое продвижение разрешается в рамках одного тика. Реализация — через разрешение намерений по освобождающимся клеткам. В MVP допустимо упрощение: занятая клетка считается недоступной в этом тике (агент ждёт один тик). Выбранный вариант фиксируется в `DECISIONS.md`.

6. **Тупик/застревание.** Если агент `N_stuck` тиков подряд не может продвинуться и не имеет валидного маршрута, он переводится в состояние `blocked`/`stuck`. Порог `N_stuck` — параметр.

**Плотность** определяется как число агентов в окрестности клетки (например, в окне `k × k` или в радиусе `r`). Плотность используется как вход в стоимость клетки для адаптивного алгоритма и как метрика.

## 9. Динамическая среда

Динамика — ключевая особенность проекта. Без неё проект вырождается в обычный статический поиск пути; с ней появляется исследовательская ценность.

**Динамические события:**
- блокировка выхода после заданного тика;
- блокировка коридора/прохода;
- появление задымления / зоны риска (повышение стоимости клеток) — Strong Version;
- изменение проходимости участка во времени — Strong Version.

**MVP-минимум:** одно событие — через заданный тик блокируется выход или проход. Этого достаточно, чтобы показать разницу между алгоритмами, которые пересчитывают маршрут, и теми, которые этого не делают.

**Strong Version:** зоны риска, задымление, изменение весов во времени, несколько событий в одном сценарии.

Каждое событие задаётся как `{тип, координаты/цель, тик_срабатывания}` и фиксируется в описании сценария, чтобы эксперимент был воспроизводим.

## 10. Алгоритмы маршрутизации

В MVP сравниваются **три** чётко различимых алгоритма. Все работают в идентичных условиях (одна карта, один seed, одни стартовые позиции — см. раздел 16).

### A1. Nearest Exit Baseline (ближайший выход)
- **Выбор выхода:** геометрически ближайший выход по манхэттенской дистанции.
- **Построение пути:** кратчайший путь до выбранного выхода с учётом стен (BFS/A*).
- **Ключевое отличие:** выход выбирается один раз по расстоянию и **не пересматривается** при перегрузке или блокировке этого выхода (пересчёт пути — только если текущий путь физически разорван). Это простой и предсказуемый baseline.
- **Слабость (намеренная):** может загнать большинство агентов в один выход; плохо реагирует на блокировку именно выбранного выхода.

### A2. Shortest Path A* (кратчайший путь)
- **Выбор выхода:** выход, до которого кратчайший путь по графу минимален (с учётом стен).
- **Построение пути:** A* по числу шагов (равные веса рёбер).
- **Ключевое отличие от A1:** учитывает структуру карты (стены), а не геометрию по прямой; при блокировке маршрута пересчитывает путь.
- **Слабость (намеренная):** не учитывает плотность и перегрузку — все идут «оптимально» в одни и те же узкие места, создавая пробки.

### A4. Adaptive Weighted A* (адаптивный взвешенный)
- **Главный исследуемый алгоритм.**
- **Построение пути:** A* по **стоимости клеток**, а не по числу шагов. Стоимость учитывает плотность, риск/задымление, перегрузку выхода; блокировка = бесконечная стоимость.
- **Динамика:** маршрут пересчитывается при наступлении заданных условий (см. раздел 11), а не каждый тик.
- **Ключевое отличие от A2:** избегает перегруженных и опасных участков, перераспределяя себя по карте динамически.
- **Слабость (намеренная):** сложнее, требует настройки коэффициентов, может быть избыточен в простых сценариях, склонен к осцилляции маршрутов при одновременном пересчёте многими агентами (предмет отдельного наблюдения).

**A3. Load-Aware Exit Distribution** (выбор выхода с учётом нагрузки) — перенесён в **Strong Version**, чтобы не размывать MVP и не дублировать A4. В Strong Version A3 принимает решение на уровне *выбора выхода* (глобальное распределение потоков), тогда как A4 адаптируется на уровне *пошагового пути*.

**Резюме различий A1/A2/A4:** A1 — фиксированный выбор выхода по геометрии; A2 — оптимальный путь по графу без учёта загрузки; A4 — путь по динамической стоимости с учётом среды и перегрузки.

## 11. Adaptive Weighted A*: формула стоимости и правила пересчёта

**Стоимость входа в клетку `c`:**

```
cost(c) = base
        + α · density(c)
        + β · danger(c)
        + γ · smoke(c)
        + δ · exitLoad(exit(c))
```

где:
- `base` — базовая стоимость прохода обычной клетки (например, 1);
- `density(c)` — число агентов в окрестности клетки `c` (плотность);
- `danger(c)` — уровень опасности клетки (опасная зона), 0 для обычной (Strong Version);
- `smoke(c)` — уровень задымления клетки, 0 для обычной (Strong Version);
- `exitLoad(exit(c))` — оценка перегрузки выхода, к которому ведёт клетка (доля агентов, нацеленных на этот выход, или длина очереди у выхода);
- `α, β, γ, δ` — настраиваемые неотрицательные коэффициенты;
- **заблокированная клетка:** `cost(c) = +∞` (исключается из поиска).

В MVP активны `base`, `α·density` и `δ·exitLoad` (плюс ∞ для блокировок). Слагаемые `danger` и `smoke` подключаются в Strong Version.

A* минимизирует суммарную стоимость пути; эвристика — манхэттенское расстояние до выхода, домноженное на `base` (допустимая, не переоценивает).

**Правила пересчёта маршрута (rerouting triggers).** Маршрут пересчитывается не каждый тик, а при выполнении любого из условий:
- появилась блокировка на текущем маршруте / следующий участок стал недоступен;
- плотность на ближайшем участке маршрута превысила порог `θ_density`;
- выбранный выход стал перегружен (`exitLoad > θ_exit`);
- прошло `K` тиков с последнего пересчёта (периодическая ревизия).

Каждый пересчёт увеличивает счётчик `reroutes` агента (это метрика и индикатор нестабильности/осцилляции).

**Замечание об осцилляции.** Если многие агенты одновременно убегают от перегруженного участка, маршруты могут «качаться» (эффект эгоистичной маршрутизации). Для смягчения предусмотреть гистерезис: переключаться только при заметном превышении порога, не пересчитывать чаще, чем раз в `K` тиков. Это явление само по себе — интересный объект наблюдения и обсуждается в результатах.

Конкретные значения `α, β, γ, δ, θ_density, θ_exit, K` подбираются эмпирически на этапе экспериментов и фиксируются в `DECISIONS.md`.

## 12. Информационное допущение

В v1 (MVP и базовая Strong Version) агент/алгоритм имеет **глобальную информацию** о карте, стенах, выходах, текущих блокировках и состоянии среды (плотность, перегрузка выходов). Это честное и явно заявленное упрощение.

Обоснование: цель проекта — сравнить *алгоритмы маршрутизации* в одинаковых условиях, а не моделировать неполноту восприятия человека. Глобальная информация делает сравнение чистым и воспроизводимым.

**Ограниченная видимость / радиус восприятия / локальная информация** — Future Work / Strong Version (расширенная). Это явно отмечается как ограничение модели в работе и как направление развития.

## 13. Метрики

Все метрики считаются программой автоматически. Субъективная оценка «на глаз» недопустима.

**Обязательные метрики (MVP):**
- общее время эвакуации (makespan — тик выхода последнего эвакуированного агента);
- среднее время эвакуации агента;
- максимальное время эвакуации агента;
- процент эвакуированных;
- число неэвакуированных / застрявших агентов;
- нагрузка на каждый выход (сколько агентов вышло через каждый выход);
- число пересчётов маршрута (суммарно и в среднем на агента);
- средняя и максимальная плотность в критических зонах.

**Желательные метрики (Strong Version):**
- средняя длина маршрута;
- число конфликтов за клетку;
- индекс перегрузки;
- эффективность относительно baseline (отношение метрик к A1);
- время вычисления алгоритма (вычислительная стоимость);
- разброс результатов по сериям запусков (стандартное отклонение, доверительный интервал).

Метрики времени интерпретируются совместно с процентом эвакуации (см. раздел 14): время без процента эвакуации вводит в заблуждение.

## 14. Обработка неэвакуированных агентов

Это методологически важный пункт: без чёткого правила метрики времени некорректны.

**Правила:**
1. **Время эвакуации** (среднее, максимальное, makespan) считается **только по фактически эвакуированным агентам**. Незаэвакуированные в эти метрики по времени не включаются.
2. **Процент эвакуированных** и **число застрявших** отчитываются **всегда и рядом** с метриками времени. Запрещено приводить время без процента эвакуации.
3. **Корректное сравнение makespan между алгоритмами** допускается только при равном (в идеале 100%) проценте эвакуации. При разном проценте сравнивается в первую очередь процент, затем — время.
4. **Штрафная метрика (опционально).** Для сводного сравнения можно ввести величину, где не вышедшему агенту приписывается время = `T_max` (горизонт). Это даёт единый «худший случай» и наказывает алгоритмы за застревание. Если используется — явно помечается как штрафная, отдельно от «чистого» среднего времени.
5. Агент в состоянии `blocked`/`stuck` на момент `T_max` считается неэвакуированным.

## 15. Источник случайности, seed и воспроизводимость

Без явного источника случайности «серия из 30 повторов» даёт 30 одинаковых чисел. Поэтому случайность определяется явно.

**Источники случайности в модели:**
- стартовые позиции агентов внутри стартовой зоны (основной источник вариативности);
- разрешение конфликтов за клетку (приоритет при одновременных намерениях);
- (опционально) разрыв ничьих при равной стоимости пути.

**Управление seed:**
- Все случайные решения берутся из одного ГПСЧ, инициализированного целым `seed`.
- Один `seed` → одна полностью детерминированная реализация (стартовые позиции + все разрешения конфликтов).
- **Парное сравнение алгоритмов:** при фиксированном `seed` генерируются одни и те же стартовые позиции, затем на них прогоняется каждый алгоритм. Это paired design — он резко повышает чистоту сравнения (алгоритмы соревнуются на идентичных входах).
- **Серия повторов:** одна серия = набор seed'ов `{s1..sn}`; для каждого seed прогоняются все алгоритмы; метрики агрегируются по seed'ам (среднее ± стандартное отклонение).

**Требования к воспроизводимости:**
- каждый запуск сохраняет полную конфигурацию `{сценарий, алгоритм, параметры, seed}` вместе с результатами;
- по сохранённой конфигурации результат воспроизводится бит-в-бит;
- результаты экспортируются в CSV/JSON.

## 16. Дизайн эксперимента

**Независимые переменные (что меняем):**
- алгоритм маршрутизации (A1 / A2 / A4);
- число агентов `N`;
- сценарий / план здания;
- тик срабатывания блокировки;
- ширина прохода / число выходов (через выбор сценария).

**Зависимые переменные (что измеряем):** метрики из раздела 13.

**Контролируемые переменные (что держим равным):** карта, набор seed'ов, горизонт `T_max`, правила движения, параметры алгоритма (кроме исследуемого).

**Принцип одного фактора.** В одной серии меняется один главный параметр, остальные фиксированы — тогда выводы интерпретируемы.

**Число повторов:** минимум 10 seed'ов на условие; желательно 30–100, если позволяет производительность.

**Статистика:** для каждого условия отчитываются среднее и стандартное отклонение по seed'ам; при сравнении алгоритмов на парных seed'ах желательно приводить разницу и, при достаточном `n`, доверительный интервал / парный тест. Минимум — mean ± std с указанием `n`.

**Примеры серий:**
- Серия 1: сравнение A1/A2/A4 на простом плане (baseline).
- Серия 2: влияние числа агентов `N` на метрики.
- Серия 3: влияние блокировки выхода (варьируется тик блокировки).
- Серия 4: влияние узкого прохода (варьируется ширина).
- Серия 5 (Strong): влияние перегрузки ближайшего выхода / распределения стартов.

## 17. Экспериментальные сценарии

Сценарии задаются как данные (JSON/код), без редактора карт в MVP.

**Обязательные (MVP):**

- **S1. Простой план.** Несколько комнат, стены, два выхода, без динамики. Цель: проверить базовую работу алгоритмов, получить baseline, подтвердить корректность.
- **S2. Узкое место.** Коридор/дверь, где образуется пробка. Цель: показать влияние плотности; продемонстрировать, что кратчайший путь ≠ самый эффективный.
- **S3. Динамическая блокировка.** Во время эвакуации блокируется выход или коридор. Цель: проверить адаптацию; показать слабость алгоритмов без пересчёта; оценить число застрявших.

**Желательные (Strong Version):**
- **S4. Зона риска / задымление.** Часть карты становится «дорогой». Цель: проверить весовой алгоритм; показать отличие кратчайшего пути от выгодного/безопасного.
- **S5. Перегрузка ближайшего выхода.** Большинство агентов стартует у одного выхода при наличии альтернативного. Цель: показать пользу распределения потоков (A3 vs A1).

## 18. Верификация и sanity-checks

Чтобы результатам можно было доверять, корректность модели проверяется автоматическими тестами:

1. **Оптимальность пути.** Один агент на пустой карте достигает выхода за число шагов, равное манхэттенскому расстоянию (при отсутствии стен). A2 и A4 на пустой карте дают путь той же длины, что BFS.
2. **Сохранение агентов.** Число агентов постоянно: агент либо в пути, либо эвакуирован, либо застрял; агенты не дублируются и не исчезают.
3. **Непроходимость стен.** Ни один агент никогда не оказывается в клетке-стене и не проходит сквозь неё.
4. **Эксклюзивность клетки.** В одной клетке никогда не находятся два агента одновременно.
5. **Детерминизм по seed.** Два прогона с одинаковыми `{сценарий, алгоритм, параметры, seed}` дают идентичные метрики.
6. **Реакция на блокировку.** После блокировки выхода ни один агент не эвакуируется через него; адаптивные/пересчитывающие алгоритмы перестраивают маршрут.

Эти проверки — часть кода (тесты ключевых алгоритмов) и одновременно аргумент на защите.

## 19. Главные результаты, которые должна выдавать программа

**Главный артефакт — кривая эвакуации:** график «доля эвакуированных от времени (тиков)» для каждого алгоритма на одних осях. Это самый информативный результат: сразу видно, какой алгоритм эвакуирует быстрее и кто застревает.

Помимо кривой:
- сводная таблица итоговых метрик по алгоритмам (раздел 13) с mean ± std по сериям;
- графики зависимости метрик от независимой переменной (например, среднее время от `N`);
- экспорт сырых результатов в CSV/JSON для внешнего анализа.

Результаты должны позволять сделать вывод вида: «при условиях X алгоритм A4 эвакуирует на Y% быстрее A2 при p<..., при условиях Z разница незначима».

## 30. Глоссарий

- **Тик** — дискретный шаг симуляции; единица модельного времени.
- **Агент** — условный человек в модели.
- **Стартовая зона** — область появления агентов.
- **Выход** — целевая клетка эвакуации.
- **Блокировка** — клетка/выход, ставшие непроходимыми во времени.
- **Плотность** — число агентов в окрестности клетки.
- **Контеншен** — конкуренция агентов за одну клетку; разрешается через seed-приоритет.
- **Makespan** — тик выхода последнего эвакуированного агента (общее время эвакуации).
- **Кривая эвакуации** — график доли эвакуированных от времени; главный результат-артефакт.
- **Reroute / пересчёт** — перестроение маршрута агентом по триггеру.
- **Seed** — целое число, инициализирующее ГПСЧ; обеспечивает детерминизм и воспроизводимость.
- **Парное сравнение (paired design)** — прогон всех алгоритмов на одинаковых стартовых позициях (один seed).
- **Baseline** — простейшая стратегия (A1), относительно которой измеряется выигрыш.

**Параметры модели (фиксируются в `DECISIONS.md` при подборе):**
`α, β, γ, δ` — коэффициенты стоимости; `θ_density, θ_exit` — пороги пересчёта; `K` — период ревизии маршрута; `N_stuck` — порог застревания; `T_max` — горизонт симуляции; `W × H` — размер сетки; `N` — число агентов.
