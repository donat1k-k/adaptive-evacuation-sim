## 2026-06-11 — Базовые решения проекта (зафиксированы на этапе E0)

Источник: `PROJECT_SPEC.md`. Здесь — сводка ключевых решений, действующих с v1.

### D1. Среда — 2D-сетка как граф
Прямоугольная сетка `W×H`: клетка = вершина, переход между соседями = ребро, стена = исключённая вершина, выход = целевая клетка. 4-связность в MVP. **Почему:** реализуемо в школьном проекте, прозрачно объясняется через графы, прямо подходит для алгоритмов поиска пути, не требует физики. (SPEC §5)

### D2. Время — дискретные тики
1 тик = синхронная попытка хода всех агентов; скорость = 1 клетка/тик; метрики в тиках (без перевода в секунды); горизонт `T_max`. **Почему:** простая и достаточная единица модельного времени для сравнения алгоритмов. (SPEC §7)

### D3. Один агент на клетку
В клетке не более одного агента. **Почему:** создаёт физическую конкуренцию за пространство — отсюда очереди, пробки, плотность. Без этого сравнение алгоритмов теряет смысл. (SPEC §8)

### D4. Конфликты за клетку — через seed/random priority
Симуляция синхронная: фаза намерений → фаза разрешения. При претензии нескольких агентов на одну клетку победитель определяется ГПСЧ с фиксированным seed; проигравшие ждут/пересчитывают. **Почему:** результат не зависит от порядка обхода агентов в коде; детерминизм по seed. (SPEC §8) Решение по «движению цепочкой» (SPEC §8 п.5) фиксируется отдельной записью на этапе E3.

### D5. Глобальная информация агента в v1
Агент/алгоритм знает карту, стены, выходы, текущие блокировки, плотность, перегрузку выходов. **Почему:** цель — сравнить алгоритмы маршрутизации в одинаковых условиях, а не моделировать неполноту восприятия. Ограниченная видимость — Future Work. (SPEC §12)

### D6. MVP без редактора карт; карты — как готовые сценарии/JSON
Сценарии задаются как типизированные данные (JSON/код). **Почему:** эксперименту нужны фиксированные, воспроизводимые, версионируемые карты; редактор — отдельный UI-проект, не нужен для научного результата. Редактор — опционально в Strong Version. (SPEC §17, PLAN §1)

### D7. Алгоритмы MVP — A1, A2, A4
A1 Nearest Exit Baseline (ближайший выход по геометрии, выход не пересматривается); A2 Shortest Path A* (оптимальный путь по графу с учётом стен); A4 Adaptive Weighted A* (путь по динамической стоимости `base + α·density + δ·exitLoad`, ∞ для блокировок). **Почему:** три чётко различимых уровня — baseline, оптимум по графу, адаптив с учётом среды. (SPEC §10–11)

### D8. A3 Load-Aware Exit Distribution — в Strong Version
A3 (выбор выхода с учётом нагрузки на уровне глобального распределения потоков) переносится в Strong Version. **Почему:** чтобы не размывать MVP и не дублировать A4 (A4 адаптируется на уровне пошагового пути, A3 — на уровне выбора выхода). (SPEC §10, §21)

### D9. Вне MVP: backend, auth, database, 3D, нейросети, CV
Не входят в MVP и базовую Strong Version. **Почему:** увеличивают риск провала, распыляют проект, не нужны для исследовательского результата; всё это Future Work. (SPEC §22) Стек: Vite + React + TS, Canvas, без backend. (SPEC §23)

### D10. Программа — исследовательский стенд, а не самоцель
Анти-цель: «красивый симулятор ради симулятора». Цель: стенд + динамическая среда + сравнение алгоритмов + метрики + эксперименты + выводы. **Почему:** этот принцип имеет приоритет при любых спорных решениях по разработке. (SPEC §4)
