## 2026-06-11 — Базовые решения проекта (зафиксированы на этапе E0)

Источник: `PROJECT_SPEC.md`. Здесь — сводка ключевых решений, действующих с v1.

### D1. Среда — 2D-сетка как граф
Прямоугольная сетка `W×H`: клетка = вершина, переход между соседями = ребро, стена = исключённая вершина, выход = целевая клетка. 4-связность в MVP. **Почему:** реализуемо в школьном проекте, прозрачно объясняется через графы, прямо подходит для алгоритмов поиска пути, не требует физики. (SPEC §5)

### D2. Время — дискретные тики
1 тик = синхронная попытка хода всех агентов; скорость = 1 клетка/тик; метрики в тиках (без перевода в секунды); горизонт `T_max`. **Почему:** простая и достаточная единица модельного времени для сравнения алгоритмов. (SPEC §7)

### D3. Один агент на клетку
В клетке не более одного агента. **Почему:** создаёт физическую конкуренцию за пространство — отсюда очереди, пробки, плотность. Без этого сравнение алгоритмов теряет смысл. (SPEC §8)

### D4. Конфликты за клетку — через seed/random priority
Симуляция синхронная: фаза намерений → фаза разрешения. При претензии нескольких агентов на одну клетку победитель определяется ГПСЧ с фиксированным seed; проигравшие ждут/пересчитывают. **Почему:** результат не зависит от порядка обхода агентов в коде; детерминизм по seed. (SPEC §8) Решение по «движению цепочкой» (SPEC §8 п.5) фиксируется отдельной записью на этапе E3.

### D5. Глобальная информация агента в v1
Агент/алгоритм знает карту, стены, выходы, текущие блокировки, плотность, перегрузку выходов. **Почему:** цель — сравнить алгоритмы маршрутизации в одинаковых условиях, а не моделировать неполноту восприятия. Ограниченная видимость — Future Work. (SPEC §12)

### D6. MVP без редактора карт; карты — как готовые сценарии/JSON
Сценарии задаются как типизированные данные (JSON/код). **Почему:** эксперименту нужны фиксированные, воспроизводимые, версионируемые карты; редактор — отдельный UI-проект, не нужен для научного результата. Редактор — опционально в Strong Version. (SPEC §17, PLAN §1)

### D7. Алгоритмы MVP — A1, A2, A4
A1 Nearest Exit Baseline (ближайший выход по геометрии, выход не пересматривается); A2 Shortest Path A* (оптимальный путь по графу с учётом стен); A4 Adaptive Weighted A* (путь по динамической стоимости `base + α·density + δ·exitLoad`, ∞ для блокировок). **Почему:** три чётко различимых уровня — baseline, оптимум по графу, адаптив с учётом среды. (SPEC §10–11)

### D8. A3 Load-Aware Exit Distribution — в Strong Version
A3 (выбор выхода с учётом нагрузки на уровне глобального распределения потоков) переносится в Strong Version. **Почему:** чтобы не размывать MVP и не дублировать A4 (A4 адаптируется на уровне пошагового пути, A3 — на уровне выбора выхода). (SPEC §10, §21)

### D9. Вне MVP: backend, auth, database, 3D, нейросети, CV
Не входят в MVP и базовую Strong Version. **Почему:** увеличивают риск провала, распыляют проект, не нужны для исследовательского результата; всё это Future Work. (SPEC §22) Стек: Vite + React + TS, Canvas, без backend. (SPEC §23)

### D10. Программа — исследовательский стенд, а не самоцель
Анти-цель: «красивый симулятор ради симулятора». Цель: стенд + динамическая среда + сравнение алгоритмов + метрики + эксперименты + выводы. **Почему:** этот принцип имеет приоритет при любых спорных решениях по разработке. (SPEC §4)

## 2026-06-11 — Решения по типам моделей (этап E2)

### D11. Конвенции типов моделей данных
- Отсутствующие значения кодируются как `| null` (например, `tEvacuated: number | null`), а не как опциональные `?:` — из-за `exactOptionalPropertyTypes: true` это явнее и безопаснее.
- Union-литералы вместо enum'ов и магических строк: `const X = [...] as const; type X = (typeof X)[number]` (CellType, AgentState, AlgorithmId, DynamicEventType). Даёт и тип, и runtime-список для итерации/валидации.
- Динамические события — дискриминированное объединение по полю `type` (`event.ts`); добавление нового типа события не ломает существующие.
- Стартовые зоны и выходы хранятся внутри `EnvironmentMap` (а не дублируются в `Scenario`), чтобы исключить рассинхронизацию.
- Сетка — row-major `cells[y][x]` (`CellType[][]`).
- Коэффициенты A4 (`AdaptiveWeights`) и пороги (`RerouteThresholds`) объявлены в `algorithm.ts`, но входят в `SimulationConfig` — алгоритм-специфичная настройка собрана в одном месте, конфиг её агрегирует.
- Модели пассивны: без runtime-логики, без `any`, без внешних библиотек. **Почему:** соответствует SPEC §25 (модели — данные, не поведение) и упрощает E3–E9.

## 2026-06-11 — Решения по движку симуляции (этап E3)

### D12. Движение цепочкой (ConflictSettings.allowChaining) — в E3 НЕ реализуется
E3 реализует MVP-упрощение SPEC §8 п.5: **без chaining** (`allowChaining=false`). Клетка, занятая на начало тика, недоступна в этом тике — претендующий агент ждёт. **Почему:** chaining требует фикс-точечного разрешения по освобождающимся клеткам (агент A входит туда, откуда уходит B) — лишняя сложность и риск недетерминизма, не нужные для демонстрации контеншена/пробок. Флаг не игнорируется: при `allowChaining=true` движок бросает явную ошибку «не реализовано в E3», а не молча делает иное. Chaining отложен (Strong / поздний этап). (SPEC §8, развивает D4)

### D13. ГПСЧ — mulberry32 (без зависимостей)
Единый seedable PRNG `mulberry32` в `src/simulation/rng.ts`. **Почему:** детерминированный, быстрый, ~3 строки, не требует внешней библиотеки (SPEC §25, PLAN §11 «без лишних зависимостей»). Используется для (1) тасовки стартовых клеток при размещении агентов и (2) приоритетов при разрешении конфликтов. Один seed → бит-идентичный прогон. Размещение агентов расходует ГПСЧ ДО любой по-тиковой случайности, поэтому стартовые позиции зависят только от `seed`+сценарий, не от алгоритма (основа парного сравнения, E9). (SPEC §15)

### D14. Временная stub-политика движения (только E3)
`StubGreedyPolicy` (`src/simulation/policy.ts`) — **временная тестовая** политика для проверки движка, НЕ финальный алгоритм. Жадный спуск по манхэттену к ближайшему открытому выходу, без поиска пути (без BFS/A*); может застревать за стенами (ожидаемо). Подаётся через интерфейс `MovementPolicy` — это шов, в который на E5 встанут A1/A2/A4. **Почему:** A* появляется только в E5, но движок нужно проверять уже сейчас на детерминизм/контеншен/завершение. (PLAN §3 E3, §3 E5)

### D15. Уточнения семантики движка (E3)
- **tEvacuated = tick + 1:** агент, попавший на выход в ходе тика `tick`, считается эвакуированным ПОСЛЕ завершения шага → `tEvacuated = tick + 1`. **Почему:** эвакуация — результат завершённого шага, а не его середины; согласованно для метрик E7.
- **Недостаток стартовых клеток → ошибка:** если уникальных клеток стартовых зон меньше `agentCount`, движок бросает понятную ошибку, а не создаёт молча меньше агентов. **Почему:** молчаливое занижение N исказило бы эксперимент.
- **`stuck`/`blocked` — диагностические, обратимые:** это состояния-индикаторы (с счётчиком `stuckTicks`), а не «смерть» агента. Агенты опрашиваются политикой каждый тик; при изменении среды снова двигаются (`moving`), `stuckTicks` сбрасывается успешным ходом. **Почему:** окончательное исключение агента делало бы реакцию на разблокировку невозможной (важно для E6).
- **Детерминированный порядок событий:** `pendingEvents` сортируются (по тику, затем по стабильному ключу type+содержимое) при инициализации. **Почему:** порядок применения не должен зависеть от порядка массива при будущем редактировании сценариев.
- **hazard-appear в E3 — базово:** копится в `state.hazards` без влияния на движение (полное влияние — Strong Version). (SPEC §9)

## 2026-06-12 — Решение по итогам аудита E3

### D16. Adjacency guard — инвариант движка (защита контракта MovementPolicy)
В фазе сбора намерений движок проверяет: политика возвращает либо `null` («стоять»), либо клетку на расстоянии `manhattan = 1` (4-сосед). Несоседняя клетка (manhattan > 1) или сама клетка агента (manhattan = 0, «сходить в себя») → `throw` с понятным сообщением. **Почему:** будущие алгоритмы (E5: A1/A2/A4) подаются через `MovementPolicy`; ошибочный ход (телепорт/«остаться не через null») должен падать громко, а не маскироваться, иначе это исказило бы движение и метрики. Это валидация инварианта (4-связность, 1 клетка/тик — SPEC §7–8), **не** реализация поиска пути. Стоять можно ТОЛЬКО через `null` — чтобы алгоритмы не путали «остаться» и «шаг в себя». (развивает D4, D14)
